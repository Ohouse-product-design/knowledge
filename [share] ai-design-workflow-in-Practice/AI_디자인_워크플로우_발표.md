# 디자인 워크플로우에서의 AI 활용기

**오늘의집 프로덕트 디자인**

Figma Make, Claude Code, MCP를 활용해 디자인 프로세스를 어떻게 바꿔나갔는지

---

## Agenda — 오늘 이야기할 것들

1. **디자인 초안 그리기** — Figma Make로 Flow 기반 프로토타입 + Figma MCP로 디자인 시스템 활용
2. **UT용 프로토타입 제작** — Claude Code + MCP로 실제 동작하는 인터랙티브 목업 제작
3. **복잡한 정책 커뮤니케이션** — 조건별 UI 상태를 직접 조작하며 확인할 수 있는 인터랙티브 스펙

---

## Case 01 — 디자인 초안 그리기

### 왜 Figma Make를 택했나

저는 디자인할 때 프로토타입을 거의 필수로 만들어요.
화면을 이어붙여 실제 동작을 직접 보면서 개선해야,
유저 입장에서 내 생각이 맞는지 판단이 되거든요.

> 단편적인 화면이 아닌 **Flow를 그려주는 AI.**
> 프로토타입 중심으로 작업하는 제게 딱 맞았습니다.

### Figma Make가 좋았던 점

- **Flow 기반 생성** — 화면 하나가 아니라 화면 간 전환과 흐름을 통째로 만들어줌. 유저 여정 전체를 빠르게 프로토타입으로 확인.
- **아이데이션 도구** — 시안 1~2개가 아니라 4~5개도 가능. 명령하면 새로운 아이디어를 만들어주니 탐색 범위가 넓어짐.
- **초안 속도** — 직접 그리면 반나절 걸릴 초안을 프롬프트 몇 번으로 뽑아냄. 방향을 빠르게 잡고 디테일에 집중.

### Figma MCP로 디자인 시스템 + 실 데이터 활용

공식 Figma MCP에 쓰기가 안 되던 시절, 오픈소스를 활용해 쓰기 가능한 Figma MCP를 직접 구성했습니다.

**가능해진 것**
- ODS 컴포넌트/토큰을 활용한 화면 제작
- ODS 토큰 미적용 영역에 토큰 일괄 적용 — 근데 직접 하는 게 더 빠를 수도?
- 화면이 많을 때 공통 수정사항 일괄 반영
- Athena MCP로 실제 상품 정보를 가져와 디자인에 적용

**현실적인 한계**
- 속도가 너무 느림 — 화면 하나에도 시간이 꽤 걸림
- ODS를 "쓸 줄 아는 것"과 "잘 그리는 것"은 다른 문제
- 결국 스틸 이미지 — 초안을 그리기에는 역부족

### Case 01 정리

> **"잘 그리는 것"보다 "많이, 빠르게 탐색하는 것"**

AI에게 완성도 높은 화면을 기대하기보다
빠르게 여러 방향을 탐색하는 도구로 쓸 때 가장 효과적이었습니다.

## Case 02 — UT용 프로토타입 제작

### 배경: Figma Make 토큰이 다 떨어졌다

우리 팀은 UT를 정말 자주 합니다.
Figma Make로 프로토타입을 만들어 UT를 진행해왔는데,
토큰이 소진되면서 대안이 필요해졌어요.

> Figma 안에서만 만들 필요가 없잖아?
> → Claude Code로 **실제 동작하는 HTML 목업**을 만들자

### 제작 과정

1. **Figma MCP로 디자인 읽기** — 기존 Figma 디자인의 구조/스타일 정보를 그대로 읽어옵니다.
2. **Claude Code로 HTML 화면 구현** — 읽어온 디자인을 기반으로 실제 동작하는 HTML을 만듭니다.
3. **인터랙션 붙이기** — 화면 전환, 버튼, 스크롤, 탭 전환 등 실제 앱과 유사한 인터랙션을 하나씩 붙여나갑니다.
4. **Athena MCP로 유저별 실 데이터 적용** — 유저별 최근 3개월 내 같은 카테고리 상품 2개 이상 조회 데이터를 가져와 목업에 반영. UT 참여자가 실제로 봤던 상품이 화면에 나오니 몰입도가 확 올라갑니다.

### Before vs After

**Before — Figma MCP로 Figma에 그리기**
- 스틸 이미지 기반 → 클릭 영역 제한적
- 데이터는 임의 텍스트로 채움
- 유저가 "이건 목업이구나" 인지 → 몰입도 한계

**After — Claude Code로 HTML 목업**
- 실제 동작하는 HTML → 스크롤, 탭, 전환 모두 작동
- Athena MCP로 유저가 실제 본 상품 데이터 적용
- "이거 진짜 앱이에요?" 반응 → 몰입도 극대화

### Live Demo — 상품비교 UT 프로토타입

Figma 디자인을 MCP로 읽어와 그대로 HTML로 구현. 스크롤, 탭 전환 등 실제 앱처럼 동작합니다.
Athena MCP로 유저가 최근 3개월 내 같은 카테고리에서 2개 이상 조회한 실제 상품 데이터를 적용했습니다.

> 데모 파일: `상품비교_mobile_light.html`

### Case 02 정리

**장점**
- Figma Make로는 불가능했던 실제 인터랙션 구현
- 유저별 실 데이터로 몰입도 극대화
- 토큰 제한 없이 원하는 만큼 수정/반복 가능
- 디자인 변경 시 프롬프트로 빠르게 반영

**한계**
- Figma 안에서 바로 안 되고 별도 HTML 파일로 관리
- 프롬프트를 잘 써야 디자인 피델리티가 올라감
- 복잡한 인터랙션은 여러 번 시행착오 필요

## Case 03 — 복잡한 정책 커뮤니케이션

### 배경: 문서로는 전달하기 어려운 복잡한 조건부 UI

커머스는 정책이 유기적으로 엮여 있어서,
문구로 설명은 쉬운데 케이스가 많고 화면 상태를 인지하기 어렵습니다.

PDP ATF 가격 영역만 해도:
`상품쿠폰` × `장바구니쿠폰` × `패키지 상품쿠폰(신규)` × `수령 여부 조합`

→ 모두 미수령 / 일부 수령 / 전부 수령... 케이스가 기하급수적으로 늘어남

### 기존 방식의 한계

텍스트 스펙 + 케이스 나열로 정리해도

> **"이 조건일 때 화면이 정확히 어떻게 되지?"** 매번 다시 물어봄

### Solution: 읽는 스펙이 아니라 만져보는 스펙

조건을 직접 토글하면 폰 목업의 UI가 실시간으로 바뀝니다.
문서를 읽고 상상하는 게 아니라, 직접 눈으로 확인하는 스펙.

1. **컨트롤 패널에서 조건 토글** — 상품쿠폰, 장바구니쿠폰, 패키지쿠폰 수령 여부를 ON/OFF
2. **실시간 UI 반영** — 쿠폰할인가, 버튼 상태, 문구가 조건에 따라 즉시 바뀜
3. **자기주도 확인** — 개발/기획/QA가 궁금한 케이스를 스스로 조작하며 확인

### Live Demo — PDP 쿠폰 영역 인터랙티브 스펙

왼쪽 패널에서 조건을 토글하면 오른쪽 목업이 실시간으로 변합니다

> 데모 파일: `pdp-coupon-prototype(B, C).html`

### Case 03 정리

**장점**
- "이 케이스일 때 어떻게 돼요?" 질문이 사라짐
- 개발/기획/QA 모두 같은 화면을 보고 대화
- 새 조건 추가 시 프롬프트 한 줄로 케이스 추가
- 스펙 문서와 달리 시각적으로 즉시 검증

**한계**
- 정책 변경 시 프로토타입도 업데이트 필요
- 모든 스펙이 아닌 조건부 UI가 복잡한 경우에 적합
- 초기 프롬프트에 정책 이해도가 필요

> **복잡한 정책일수록 문서 < 직접 만져보는 것**

## Wrap-up — AI 활용의 핵심은 "어디에 쓸 것인가"

1. **초안은 빠르게** — Figma Make로 Flow 기반 시안을 여러 개 빠르게 뽑아내고 아이데이션 도구로 활용
2. **UT는 몰입감 있게** — Claude Code + 실 데이터로 유저가 "진짜 앱"이라고 느끼는 프로토타입 제작
3. **커뮤니케이션은 직관적으로** — 복잡한 정책을 인터랙티브 프로토타입으로 만들어 직접 만져보게 하기

AI가 디자인을 대신하는 게 아니라

> **디자이너의 판단을 더 빠르고 정확하게 검증**할 수 있도록 도와주는 도구
