Route 02🎯 정밀

Figma MCP + Claude Code 정밀 루트

핵심 — MCP를 연결하면 AI가 피그마를 더 정확하게 읽음.

① 기본 프롬프트

@피그마링크@ 이 화면을 똑같이 html로 만들어줘. Figma MCP로 화면 읽어서 구조를 정확하게 파악해줘.

② 하나씩 디벨롭

상단 탑바는 sticky 되게 해줘 @이 영역@은 버튼을 눌렀을 때 아래로 펼쳐지게 해줘 카드를 클릭하면 상세 화면으로 이동하는 애니메이션을 넣어줘
💡 @이 영역@처럼 구체적으로 지칭하는 게 핵심. "저기 위에" 같은 모호한 표현은 AI가 잘못 이해합니다.

가장 시간이 오래 걸리는 단계 — 7가지 기법을 섞어 쓰자

Principles

모든 방식에 통하는 3가지 원칙

PRINCIPLE 01
잘게 쪼개기
전체 재생성 ❌ · 깨진 부분만 선택해서 재요청
PRINCIPLE 02
수치 무조건 명시
AI의 상상력을 믿지 말 것 · 숫자·위치·크기 모두 명시
PRINCIPLE 03
변경 전 상태 저장
git commit으로 체크포인트 만들기 · 망가지면 복원

수치 명시 — 나쁜 예 vs 좋은 예

❌ 나쁜 예
카드를 여러 개 배치해줘
✅ 좋은 예
카드를 정확히 6개 배치해줘. - 그리드: 2열 x 3행 - 카드 간 간격: 가로 16px, 세로 24px - 카드 높이: 180px 고정
Technique 01

QA 스타일 번호 매기기

수정사항을 1·2·3 번호로 매겨서 한 번에 전달. AI가 누락 없이 처리.

아래 수정사항을 반영해줘. (첨부 이미지 참고) 1. Radius 12로 변경 2. 텍스트 값 semibold인지 확인 및 변경 필요 3. 카드 간격 16px → 24px 4. 하단 CTA 버튼 full width로
✅ STRENGTH
누락 없음
AI가 번호 매겨진 지시를 빠뜨리지 않음
🎯 STRENGTH
정확한 재확인
"1번은 됐는데 3번이 안 됐어" 식으로 특정 가능
💬 STRENGTH
소통 효율
개발자에게 QA 시트 전달하듯 명확함
Technique 02

이미지로 잘게 잘라 재전달

수치를 말로 전달하기 어려울 때. 수정 필요 영역만 캡처.

기본 수정 요청

이 이미지랑 동일하게 다시 맞춰줘. 칩 영역이 2열 그리드로 나와야 하고, 버튼은 하단에 full width로.

AS-IS / TO-BE 패턴 (디자인 변경 시)

Flow는 동일해. 아래 두 이미지 참고해서 디자인 변경해줘. [AS-IS 이미지 첨부] [TO-BE 이미지 첨부] 이 이미지에서 이 이미지로 디자인이 변경되었으니 프로토타입에서도 동일하게 변경해줘.
Technique 03

프롬프트 오브 프롬프트 ⚡

말로 설명하기 어려운 인터랙션(바텀시트·복잡한 트랜지션)을 요청할 때.

💡 핵심 아이디어 — "어떻게 설명해야 AI가 알아들을까"를 또 다른 AI에게 물어보는 것.

Step 1. Claude에게 프롬프트 생성 요청

네이버 지도 앱 바텀시트 같은 걸 만들고 싶어. 아래에서 위로 끌어올릴 수 있고, 세 단계(닫힘/반/열림)로 고정되고, 끌어올리면 검색 결과 리스트가 나오는 구조야. 이걸 Cursor AI에게 정확하게 구현시킬 수 있도록 상세한 프롬프트로 변환해서 제안해줘. position, z-index, transition 같은 세부 스펙도 포함해서.

→ AI가 만든 상세 프롬프트를 Cursor에 그대로 복붙.

A/B/C 시안 만드는 3가지 접근법

접근법 방식 생성 속도 UT 적합성
A. 프롬프트형 "ABCD 시안 4가지 제안해줘" ⚡ 빠름 ⚠️ 각각 따로 전달
B. 베이스 복제형 시작화면에 A/B/C 버튼 분기 🐢 느림 ✅ 시작 메뉴 통합
C. 제약 풀기형 "꼭 ~가 아니어도 돼" 힌트 ⚡ 보통 ⚠️ 후속 정리 필요
Approach B

베이스 복제형 — 가장 추천 ⭐⭐

베이스 프로토타입을 하나 만들고 → 시작화면(메뉴)에서 A/B/C안으로 분기.

첫 시작화면을 하나 만들어줘. 그리고 시작화면에 A안이라고 버튼을 만들고 그걸 누르면 현재 만들어진 프로토타입이 나오게 해줘. 그리고 첫 시작화면에 B안과 C안 버튼도 추가해줘. B안과 C안도 A안과 동일하게 우선 작동하도록 만들어줘. 이어서 B안과 C안을 개선할게.

각 안 개선 예시

B안에서만 상단 네비게이션을 숨기고, 대신 플로팅 버튼을 우측 하단에 추가해줘. C안에서만 카드 리스트를 그리드가 아니라 세로 리스트로 바꿔줘.
Approach C

제약 풀기형 ⭐

이미 만든 초안에서 다른 방향으로 변형할 때. "꼭 ~가 아니어도 돼"가 핵심 키워드.

이 화면은 포인트 적립 안내를 보여주는 화면이야. 사용자가 해당 페이지에 진입했을 때 얼마의 포인트를 받을 수 있는지 시각적으로 강조해야 해. 나는 관련된 아이콘과 실제 적립되는 포인트를 함께 보여주는 시안을 배너 형태로 만들었어. UI가 꼭 배너 형태가 아니어도 돼. UI 형태, 위치 등 다양하게 고려해서 새로운 시안들을 만들어줘.
🔑 "꼭 ~가 아니어도 돼" — AI에게 "기존 형식 제약을 풀어도 된다"는 신호를 전달.

Pitfalls

AI가 망치는 3가지 패턴

실무자들이 공통으로 겪는 함정. 대응법을 외워두세요.

PITFALL 01
복잡한 인터랙션 꼬임
증상 — 바텀시트 z-index 뒤집힘, 스크롤 오작동
대응 — 깨진 부분만 캡처 재전달 + 프롬프트 단계 분할
PITFALL 02
수치 임의 채움
증상 — 스펙에 없는 숫자를 상상으로 채움
대응 — 모든 수치 명시 + 결과 반드시 검증
PITFALL 03
수정 누적 붕괴
증상 — 반복 수정 중 엉뚱한 곳이 깨짐
대응 — git commit 체크포인트 + 변경 범위 좁게
💾 빠른 되돌리기 프롬프트 — 방금 전 변경 취소해줘. 이전 상태로 되돌려줘.

Takeaways

오늘 기억할 5가지

01
완성도보다 검증 속도
목적은 UT 검증이지 피그마 복제가 아니다. 핵심 화면·인터랙션에 집중.
02
잘게 쪼개서 반복
전체 재생성은 망하는 지름길. 깨진 부분만 선택해서 재요청.
03
수치는 무조건 명시
AI의 상상력을 믿지 말 것. 숫자·위치·크기를 모두 명시.
04
내 도구 조합을 찾아라
6가지 루트가 모두 달랐다. 정답은 없고 내 과업에 맞는 조합만 있다.
05
재사용 자산을 쌓아라
같은 구조는 패턴으로 문서화. 프로젝트가 쌓일수록 빨라지는 구조로.
