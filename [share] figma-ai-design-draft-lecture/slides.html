✋ PRACTICE 01 · 15분

첫 프로토타입 만들어보기

목표

경량 루트로 피그마 화면 하나를 HTML 프로토타입으로 변환.

준비물

  • Claude Code 설치 완료
  • 피그마 계정 · 공유받은 프레임 링크

단계

  1. 작업 폴더 만들기 — mkdir my-first-proto
  2. 터미널에 claude 입력
  3. 프롬프트 복붙 + URL 교체
  4. 브라우저에서 localhost:3000 확인
  5. 요소 하나 수정 요청 — 예: "헤더 배경색 흰색으로"

🎯 체크포인트

브라우저 결과가 원본 피그마와 70% 이상 닮았는가?

그렇다면 성공. 아니면 URL을 다시 확인하거나 프레임이 복잡하지 않은지 점검.

💡 팁

첫 시도에서 완벽히 나오지 않는 것은 정상. 원칙 "하나씩 수정"을 따라 한 번에 한 요소씩 고치세요.

TECHNIQUE 01

QA 스타일 번호 매기기

수정사항을 1·2·3 번호로 매겨서 한 번에 전달. AI가 누락 없이 처리.

아래 수정사항을 반영해줘. (첨부 이미지 참고)

1. Radius 12로 변경
2. 텍스트 값 semibold인지 확인 및 변경 필요
3. 카드 간격 16px → 24px
4. 하단 CTA 버튼 full width로

✅ 누락 없음

AI가 번호 매겨진 지시를 빠뜨리지 않음

🎯 정확한 재확인

"1번은 됐는데 3번이 안 됐어" 식으로 특정 가능

💬 소통 효율

개발자에게 QA 시트 전달하듯 명확함

TECHNIQUE 06

분홍 화살표 플로우 연결 ⭐

가장 창의적인 기법. 피그마에서 화면 간 이동을 분홍 화살표로 그려두고 그 플로우대로 프로토타입 생성.

사전 준비 (피그마)

  1. 여러 화면을 피그마 캔버스에 배치
  2. 화면 A의 버튼 → 화면 B로 이동 관계를 분홍색 화살표로 그리기
  3. 모든 화면을 하나의 Frame으로 묶기
  4. Frame 전체 선택 → Copy link to selection
이 링크 타고가면 화면별로 어떻게 동작하는지
분홍색 화살표로 flow 연결해뒀는데
그 순서대로 프로토타입 만들어줘.

링크: (Frame 전체 링크)
"놀랍게도 잘 만들어줌" — 실무자 후기

VARIATIONS

A/B/C 시안 만드는 3가지 접근법

접근법 방식 생성 속도 UT 적합성
A. 프롬프트형 "ABCD 시안 4가지 제안해줘" ⚡ 빠름 ⚠️ 각각 따로 전달
B. 베이스 복제형 시작화면에 A/B/C 버튼 분기 🐢 느림 ✅ 시작 메뉴 통합
C. 제약 풀기형 "꼭 ~가 아니어도 돼" 힌트 ⚡ 보통 ⚠️ 후속 정리 필요

APPROACH B

베이스 복제형 ⭐⭐ 가장 추천

베이스 프로토타입을 하나 만들고 → 시작화면(메뉴)에서 A/B/C안으로 분기.

첫 시작화면을 하나 만들어줘.
그리고 시작화면에 A안이라고 버튼을 만들고
그걸 누르면 현재 만들어진 프로토타입이 나오게 해줘.

그리고 첫 시작화면에 B안과 C안 버튼도 추가해줘.
B안과 C안도 A안과 동일하게 우선 작동하도록 만들어줘.

이어서 B안과 C안을 개선할게.

각 안 개선 예시

B안에서만 상단 네비게이션을 숨기고,
대신 플로팅 버튼을 우측 하단에 추가해줘.

C안에서만 카드 리스트를 그리드가 아니라 세로 리스트로 바꿔줘.
30 / 40

APPROACH C

제약 풀기형 ⭐

이미 만든 초안에서 다른 방향으로 변형할 때. "꼭 ~가 아니어도 돼"가 핵심 키워드.

  1. 화면의 배경·목적 설명
  2. 현재 내가 만든 시안 설명
  3. 제약 풀기 프롬프트 전달
이 화면은 포인트 적립 안내를 보여주는 화면이야.
사용자가 해당 페이지에 진입했을 때
얼마의 포인트를 받을 수 있는지 시각적으로 강조해야 해.

나는 관련된 아이콘과 실제 적립되는 포인트를
함께 보여주는 시안을 배너 형태로 만들었어.

UI가 꼭 배너 형태가 아니어도 돼.
UI 형태, 위치 등 다양하게 고려해서 새로운 시안들을 만들어줘.
🔑 "꼭 ~가 아니어도 돼" — AI에게 "기존 형식 제약을 풀어도 된다"는 신호를 전달

목표

실습 1·2에서 만든 프로토타입을 베이스로, A/B/C 세 가지 시안을 만들어본다.

미션

PITFALLS

AI가 망치는 3가지 패턴

실무자들이 공통으로 겪는 함정. 대응법을 외워두세요.

01

복잡한 인터랙션 꼬임

증상: 바텀시트 z-index 뒤집힘, 스크롤 오작동

대응: 깨진 부분만 캡처 재전달 + 프롬프트 단계 분할

02

수치 임의 채움

증상: 스펙에 없는 숫자를 AI가 상상으로 채움

대응: 모든 수치 명시 + 결과 반드시 검증

03

수정 누적 붕괴

증상: 반복 수정 중 엉뚱한 곳이 깨짐

대응: git commit 체크포인트 + 변경 범위 좁게

TAKEAWAYS

오늘 기억할 5가지

  1. 완성도보다 검증 속도
    목적은 UT 검증이지 피그마 복제가 아니다. 핵심 화면·인터랙션에 집중.
  2. 잘게 쪼개서 반복
    전체 재생성은 망하는 지름길. 깨진 부분만 선택해서 재요청.
  3. 수치는 무조건 명시
    AI의 상상력을 믿지 말 것. 숫자·위치·크기 모두 명시.
  4. 내 도구 조합을 찾아라
    6가지 루트가 모두 달랐다. 정답은 없고 내 과업에 맞는 조합만 있다.
  5. 재사용 자산을 쌓아라
    같은 구조는 패턴으로 문서화. 프로젝트가 쌓일수록 빨라지는 구조로.
